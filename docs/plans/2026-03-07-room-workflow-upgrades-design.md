# 房间流程与体验升级设计

## 背景

当前项目已经具备房间、买入、审核、结算、十三水对局、座位抽签工具等基础能力，但在真实使用流程里还存在几个明显断点：

1. 用户登录后不会自动恢复，重新打开 App 需要再次登录。
2. 用户名不能修改，导致长期使用不方便。
3. 房主离开房间后，用户新提交的待审核购买请求虽然仍在数据库里，但房主重新进入后不一定能第一时间看见，体验上像“消息丢了”。
4. 房主不能直接替其他玩家录入买入或结账。
5. 当前创建房间与买入流程围绕“最低/最高买入积分”设计，不符合“按手数买入”的真实使用习惯。
6. 房间内已有座位分配工具，但还没有形成“分配结果同步到房间、玩家再自己报座位号确认”的闭环。
7. App 没有安装到桌面的能力提示，也缺少 PWA 基础设施。
8. 一些页面仍存在移动端适配与 UI 交互问题。

本次改造采用“一个问题完成并验证后，再处理下一个问题”的顺序推进，优先保证业务闭环和可验证性，而不是一次性大范围重构。

## 目标

本轮升级目标分为 8 个问题，按以下顺序执行：

1. 登录后保存用户信息，下次自动登录。
2. 用户可以修改用户名。
3. 进入房间时主动恢复并显示待审核购买列表。
4. 房主可以替用户代购买、代结账，且免审核。
5. 创建房间时配置“每手积分额度”，房间内按手数购买并换算积分。
6. 房间内集成座位分配工具，分配结果同步到房间后，玩家自行报座位号确认。
7. 打开 App 时优先提供系统安装提示，并补齐 PWA 基础能力。
8. 基于最新设计技能包，修复适配和交互问题。

## 非目标

- 不在本轮引入完整账号认证体系（密码、短信、Token、RLS 改造）。
- 不重做已有德州/十三水核心结算模型。
- 不把所有临时页面状态都做本地持久化，只持久化用户身份与安装提示等必要信息。
- 不把“房间内昵称”和“全局用户名”并行维护；本轮只做全局用户名修改。

## 总体方案

### 1. 账户层

账户层负责解决“自动登录”和“用户名修改”。

- `UserContext` 从“纯内存”升级为“内存 + localStorage 恢复”。
- App 启动时先尝试恢复本地用户，再渲染受保护路由，避免启动瞬间错误跳转到登录页。
- 新增用户改名接口，修改 `users.username`，并在成功后同步更新前端上下文与本地存储。
- 由于头像基于 `username` 生成，改名后头像会自然变化，这视为符合预期。

### 2. 财务审核与买入层

财务层负责解决“待审核购买恢复”“房主代操作”“按手数购买”。

- 进入房间后，前端必须主动拉取当前 `pending_buyins` 列表，而不能仅依赖实时订阅。
- 房间时间线/购买记录统一整合 4 类事件：`待审核`、`已买入`、`已结账`、`房主代录`。
- 后端新增或扩展接口，支持房主替任意玩家直接买入和结账，并在服务端校验房主身份。
- 创建房间时移除“最低/最高买入积分”，改为单一配置 `points_per_hand`。
- 买入弹窗改为输入手数，系统自动换算出实际积分金额并落库。
- 为了保留审计能力，买入记录除了 `amount` 外，还应记录 `hand_count` 与当时的 `points_per_hand` 快照。

### 3. 座位层

座位层负责解决“房间内分配座位后同步回房间”的闭环。

- 在房间内提供进入座位分配工具的入口。
- 当工具从房间上下文进入时，点击“确定”后把分配结果回写到当前房间，而不是只在工具页本地展示。
- 房间玩家列表展示系统分配的座位号，以及每个玩家是否已“报座位号/确认座位”。
- 分配结果同步后，玩家仍需自己点击一次确认，表示“我认领这个座位”。

### 4. 安装与体验层

安装与体验层负责解决“保存到桌面”和“适配交互优化”。

- 补齐 PWA 基础：manifest、service worker、安装提示监听。
- 支持浏览器时使用系统安装提示；不支持时降级为引导弹层。
- 对主要移动端页面做定向修复，而不是盲目整体换皮。
- 重点关注底部安全区、弹层层级、点击热区、十三水桌面布局、长列表与横向滚动冲突。

## 数据模型设计

### 用户

- 继续使用 `users(id, username, created_at)`。
- 新增用户名修改接口，不新增 display_name。

### 房间配置

普通房创建配置新增/替换：

- `points_per_hand INTEGER NOT NULL DEFAULT 100`

对应变化：

- 废弃 `min_buyin`
- 废弃 `max_buyin`

说明：历史数据可继续保留旧字段以兼容旧接口，但前后端主流程不再依赖它们。

### 买入记录

`buy_ins` 增强字段：

- `hand_count INTEGER`：本次对应几手，可为空（兼容旧数据）
- `points_per_hand INTEGER`：当次换算使用的每手积分，可为空（兼容旧数据）
- `created_by UUID`：由谁录入；自己买入时等于 `user_id`，房主代录时为房主 ID
- `source TEXT`：`self` / `host_proxy` / `approval`

### 待审核买入

`pending_buyins` 增强字段：

- `hand_count INTEGER NOT NULL`
- `points_per_hand INTEGER NOT NULL`

说明：待审核列表本身已经持久化在库中，问题本质不在“是否保存”，而在“房主进入房间时未主动拉取并展示”。

### 座位信息

`game_players` 增强字段：

- `seat_number INTEGER`：系统分配的座位号
- `seat_confirmed BOOLEAN NOT NULL DEFAULT FALSE`：玩家是否已认领/报座位号
- `seat_confirmed_at TIMESTAMPTZ`

如果一个房间重新分配座位，则：

- 覆盖 `seat_number`
- 批量重置 `seat_confirmed = false`
- 玩家需要重新确认

## 接口设计

### 1. 自动登录

无需新增后端接口。

前端启动时：

- 从 `localStorage` 读取用户信息
- 恢复到 `UserContext`
- 若数据损坏则清理并回到未登录态

### 2. 用户改名

- `PATCH /api/users/:id/username`
- body: `{ username: string }`

服务端校验：

- 非空
- trim 后长度 1~20
- 唯一性检查（保持现有全局唯一语义）

### 3. 进入房间恢复待审核列表

沿用现有：

- `GET /api/buyin/pending/:gameId`

前端在 `GameRoom` 初次加载时主动请求一次，再由实时订阅继续增量刷新。

### 4. 房主代购买

扩展现有买入接口，或者新增单独接口。推荐：

- `POST /api/buyin`
- body: `{ gameId, userId, amount, type, createdBy, handCount, pointsPerHand, source }`

规则：

- 普通用户自购：`createdBy === userId`, `source = self`
- 待审核通过落库：`source = approval`
- 房主代购：`createdBy = hostId`, `userId = targetUserId`, `source = host_proxy`

服务端必须校验：

- 若 `createdBy !== userId`，则 `createdBy` 必须是房主

### 5. 房主代结账

扩展：

- `POST /api/buyin/checkout`
- body: `{ gameId, userId, chips, createdBy }`

规则：

- 自己结账：`createdBy === userId`
- 房主代结账：`createdBy = hostId`, `userId = targetUserId`

### 6. 房间按手数购买

创建房间：

- `POST /api/games`
- body 用 `pointsPerHand` 取代现有 `minBuyin/maxBuyin` 在前端主流程中的作用

买入时：

- 前端输入 `handCount`
- 自动计算 `amount = handCount * pointsPerHand`

### 7. 座位分配与确认

建议新增：

- `POST /api/games/:id/seat-assign`
  - body: `{ assignments: Array<{ userId: string; seatNumber: number }>, assignedBy: string }`
- `POST /api/games/:id/seat-confirm`
  - body: `{ userId: string }`

规则：

- 只有房主能提交分配结果
- 分配后批量重置确认状态
- 玩家只能确认自己的座位

## 前端交互设计

### 登录持久化

- 首次登录成功后显示行为不变。
- 下次打开时若本地存在用户信息，直接进入上次可访问流程。
- 提供退出登录时清理本地用户缓存。

### 改用户名

- 入口放在个人中心，后续可再扩展到房间内。
- 使用轻量弹窗编辑。
- 成功后立即刷新当前页面上的用户名显示。

### 房间待审核购买列表

- 房主重新进入房间后，待审核区应立即显示库里现存未处理请求。
- 不再把“是否在线时收到消息”作为是否可见的前提。

### 代购买 / 代结账

- 房主点击玩家头像或玩家卡片，出现房主专属操作入口。
- 操作面板中可以选择：`代购买`、`代结账`。
- 代购买默认直接落库，不走审核。

### 按手数购买

- 创建房间页：显示 `每手积分额度` 输入框。
- 买入弹窗：输入框主单位改为“手”。
- 同时实时显示换算结果：
  - `3 手 = 300 分`

### 座位工具

- 房间页新增“分配座位”入口。
- 若从房间内打开座位工具，确认后直接回写房间并返回房间页。
- 房间列表展示：
  - `1 号位 - 已确认`
  - `2 号位 - 待认领`

### 安装提示

- 首次满足安装条件时弹出轻量提示。
- 用户点击“安装”时优先触发系统安装提示。
- 若系统不支持，则展示对应平台说明。

## 错误处理

- 本地缓存损坏：自动清除并回到登录页。
- 用户改名冲突：明确提示“用户名已存在”。
- 房主代操作失败：提示权限不足或房间状态异常。
- 待审核列表加载失败：显示错误 toast，但不阻塞房间主体。
- 座位分配冲突：要求重新同步最新玩家列表后再提交。
- PWA 安装提示不可用：优雅降级，不报错。

## 验证策略

每个问题完成后立即验证，再进入下一个问题。

### 问题 6：登录持久化

- 登录一次
- 刷新页面仍保持登录
- 关闭浏览器重开仍能自动登录
- 退出登录后本地缓存被清除

### 问题 3：用户名修改

- 改名成功后个人中心、大厅、房间内名称同步更新
- 本地缓存同步为新用户名
- 重启后仍为新用户名

### 问题 1：审核列表恢复

- 用户提交待审核购买
- 房主离开房间
- 房主重新进入房间
- 待审核列表仍然可见并可审批

### 问题 2 / 4 / 5：财务能力

- 房主可替玩家直接买入与结账
- 代购买不进入待审核
- 按手数购买换算正确
- 记录中能看出手数和积分

### 问题 7：座位同步

- 从房间内进入座位工具
- 分配完成点击确定后返回房间
- 房间内座位数据立即同步
- 玩家可逐个确认自己的座位

### 问题 8 / 9：安装与 UI

- 支持浏览器可弹安装提示
- 不支持浏览器能显示引导
- 主要房间页在手机尺寸下无明显遮挡、溢出、难点按钮

## 风险与取舍

### 1. 改用户名仍保持唯一

虽然用户之前在“房间内昵称”场景中允许不唯一，但当前这轮需求明确为“用户可以修改用户名”，而项目现有登录语义完全依赖 `username`。为了不破坏现有登录模型，本轮仍保持全局唯一。

### 2. 不彻底删除旧买入字段

数据库里已有 `min_buyin/max_buyin` 与相关历史逻辑。为降低迁移风险，本轮优先做到“前端主流程弃用，后端兼容保留”，而不是马上做破坏式清理。

### 3. 不保存所有运行时数据

“保存用户一切数据”如果按字面做，会把房间临时态、草稿、弹窗状态都写本地，风险很高。本轮解释为“保存用户身份和必要配置”，不扩展到整个运行内存镜像。

### 4. 安装到桌面不能真正强制

浏览器不允许网页绕过系统完成安装，因此目标是“尽量一键化触发系统提示 + 兼容平台引导”，而不是无权限的强制安装。

## 推荐执行顺序

1. 登录后保存用户信息
2. 用户修改用户名
3. 进入房间恢复待审核购买列表
4. 房主代购买 / 代结账免审核
5. 创建房间配置每手积分额度，房间内按手数购买
6. 房间内分配座位并同步、玩家自行报座位号
7. PWA 安装提示
8. UI 适配与交互优化

整个执行过程中，每完成一步都先做类型检查、构建验证和场景验证，再进入下一步。
